1. Architecture Overview
TransitOps is a decoupled client-server web application:
+-------------------+ HTTPS/JSON (JWT auth) +--------------------------+
| Next.js Client | <-----------------------------------> | Django + DRF API |
| (App Router, TS) | | (business rules, RBAC) |
+-------------------+ +-----------+--------------+
|
+-----------v--------------+
| PostgreSQL (Django ORM)|
+-----------+--------------+
|
+---------------------------v---------------------------+
| Celery worker + beat (Redis broker) |
| - License/document expiry scan (daily) |
| - Mileage rolling-average recompute + deviation alert |
| - Depot-return-overdue scan |
+---------------------------------------------------------+
Why this shape: - Next.js handles all rendering/UX and calls the API purely over REST — keeps frontend and backend
independently deployable/testable, important when a hackathon team splits frontend/backend work in parallel. - Django
+ DRF centralizes all business-rule enforcement server-side (never trust client-side validation alone) — critical since the
PS explicitly grades on business rule correctness. - Celery is isolated to the “smart” features (reminders, anomaly
detection) so the core CRUD/trip flow has zero dependency on it — if Celery/Redis setup runs out of time, it can be
dropped without breaking the Must-have MVP (the same logic can run synchronously as a fallback, see §9.3).

3.2 Key Relationships
One Vehicle → many FuelLogs, Expenses, MaintenanceRecords, VehicleDocuments, Tyres
One Trip → one Vehicle, one Driver (at a time), many Expenses, one auto-generated FuelLog on completion
One Tyre → many TyrePositionHistory rows (append-only ledger, not overwritten, so km-per-position is fully
reconstructable)
3.3 Derived / Computed Fields (not stored, computed at read time or via
signal)
Vehicle.total_operational_cost = SUM(FuelLog.cost) + SUM(MaintenanceRecord.cost) [+ SUM(Expense.amount)]
Vehicle.fuel_efficiency_kmpl = Distance / Fuel over a period
Vehicle.roi = (Revenue − (Maintenance + Fuel)) / Acquisition Cost
Vehicle.rolling_avg_mileage = average of computed_mileage_kmpl over last N fuel logs
Fleet.utilization_pct = Active Trips vehicles / Total non-retired vehicles

6. State Machines
6.1 Trip Lifecycle
Draft → Dispatched → Completed Draft → Cancelled (or) Dispatched → Cancelled
Guard conditions: - Draft → Dispatched: vehicle & driver both Available, cargo_weight ≤ vehicle.max_load_capacity, driver
license valid & not Suspended - Dispatched → Completed: requires final_odometer_km and fuel_consumed_l input -
Dispatched → Cancelled: restores vehicle & driver to Available
6.2 Vehicle Status
Available ⇄ On Trip (via trip dispatch/complete/cancel) Available ⇄ In Shop (via maintenance open/close) Available/In
Shop → Retired (manual, terminal — Retired never re-enters Available automatically)
6.3 Driver Status
Available ⇄ On Trip (via trip dispatch/complete/cancel) Available ⇄ Off Duty (manual toggle) Any → Suspended (Safety
Officer action; blocks trip assignment until lifted)
6.4 Maintenance Status
In Shop → Completed (closing triggers vehicle status restore unless Retired)
6.5 Tyre Lifecycle (stretch)
Active (position A) → Active (position B) via reposition event (closes km ledger on position A, opens on B) Active →
Retired/Worn when cumulative_km crosses configured threshold or manually marked worn
7. Business Rule Enforcement Design
All rules from PRD §7 are implemented as a service layer (services/trip_service.py, services/vehicle_service.py, etc.)
sitting between DRF views and the ORM — views never mutate model state directly. Each service function:
1. Loads the relevant objects inside a DB transaction (select_for_update() on Vehicle/Driver rows to prevent race
conditions on double-dispatch).
2. Validates preconditions (status, capacity, license expiry) and raises a domain-specific exception with a user-facing
message on failure.
3. Performs the state transition + any derived writes (e.g., dispatch creates status change + audit log row) atomically.
This guarantees the “double-booking” and “overload” rules can’t be bypassed even under concurrent requests — a
common hackathon-demo failure mode.

9. Supporting Design Details
9.1 Mileage Deviation Alert — Calculation
mileage_this_fill = (odometer_at_fill - previous_odometer) / litres
rolling_avg = average(last N computed_mileage_kmpl for this vehicle) # N = 5, configurable
if mileage_this_fill < rolling_avg * (1 - threshold): # threshold e.g. 0.15
flag FuelLog.is_anomaly_flagged = True
create Notification(type=Mileage Deviation, vehicle, message)
9.2 Vehicle ROI Formula
ROI = (Revenue − (Maintenance_cost + Fuel_cost)) / Acquisition_cost
Revenue is entered/estimated per trip (freight revenue field) or aggregated per period for the Reports screen —
documented as a manual input field on Trip if no billing module exists in MVP scope.
9.3 Fallback for Background Jobs
If Celery/Redis setup is skipped to save time, the same expiry-scan and mileage-recompute logic can run as a Django
management command triggered on dashboard load (synchronous, cached for 5 minutes) — functionally identical for
demo purposes, just not truly “scheduled.”
10. Security Design
Passwords hashed via Django’s default PBKDF2 (or Argon2 if django-argon2 is added)
JWT access token short-lived (15 min), refresh token longer-lived (7 days), via djangorestframework-simplejwt
All endpoints require authentication except /auth/login
Permission classes per view enforce the RBAC matrix (custom IsRole('Fleet Manager') style permission classes)
CORS restricted to the Next.js frontend origin only
Audit fields (created_by, updated_by, timestamps) on all mutation-heavy models (Trip, MaintenanceRecord, Vehicle
status changes)
